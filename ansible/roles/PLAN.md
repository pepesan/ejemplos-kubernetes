# 🧩 Roadmap and Status of the Roles

Backlog and status of the reusable Ansible roles in `ansible/roles/`. See [`../base/PLAN.md`](../base/PLAN.md)
for the base labs' roadmap, or [`../PLAN.md`](../PLAN.md) for the repository overview.

After completing the 14 base labs (01-14), a fair amount of duplicated or near-identical logic between
them would fit as standalone Ansible roles. The existing labs have NOT been migrated to use these roles
yet (explicit decision: build the roles standalone first, migrate later).

## Status

- ✅ **`lxd_machine_provision`** (implemented and validated with real Molecule, 2026-07-18): provisions LXD instances — **VMs or containers, indistinctly** (`lxd_instance_type`, per group or per host) — with given specs (CPU/RAM/disk), independent of whatever gets installed inside afterwards. Faithfully reproduces `02_crear_nodos.yml` (creation, waiting for the LXD agent, injecting the host's SSH key, waiting for SSH), generalizing the instance type (the original only ever created VMs). A full `molecule test` (syntax, converge, native `idempotence`, verify, destroy) passes in both scenarios (`default`=VM, `container`), leaving no residue.
- 🟢 **`lxd_host_bootstrap`** (implemented and validated across every supported distro, 2026-07-18): prepares the physical host (apt packages, Galaxy collections, kernel modules, LXD/kubectl/helm via snap, `lxd init` for network+storage+profile, base image). Faithfully reproduces `ansible/base/00_bootstrap_host_lxd.yml`, reorganized into separate task files with the hardcoded values turned into `defaults/main.yml`. Real bugs found and fixed:
  1. `lxd_host_user` and the tasks that relied on "whoever connects" (`become: false`) assumed the play is invoked as a regular user escalating with `become: true` — assuming in tests that the whole process already runs as root (`sudo -E molecule test`, no interactive password prompt) broke that assumption, and the SSH key/Galaxy collections ended up under `/root` instead of the real user. Fixed by resolving `lxd_host_user` from `SUDO_USER` (falling back to the connecting user) and explicitly targeting those tasks at it via `become_user`.
  2. The classic snap mount symlink (`/snap` → `/var/lib/snapd/snap`, needed on Debian, already present on Ubuntu) failed on **every** distro, not just Debian: `/var/lib/snapd/snap` doesn't exist yet immediately after installing the `snapd` package (only snapd itself creates it, on first run), and `ansible.builtin.file`'s `link` state refuses to create a symlink to a target that doesn't exist yet unless `force: true` is set.
  3. Loading a kernel module is inherently a host/VM-level concern — a container shares its host's kernel, and the `modprobe` executable itself may not even be present in a minimal container image (missing on one Debian 12 test container, while it happened to succeed on the others only because that module was already loaded at the host level and visible inside the container). Tagged `requires_virtualization` (same tag as `lxd_init`/`base_image`) so a container-based test can skip it.
  4. The Ansible Galaxy collections task assumes `ansible-galaxy` is already present on the target — trivially true for this role's real usage (`connection: local`, the same machine already running `ansible-playbook` to get this far), but not for a genuinely separate/fresh remote target. Tagged `requires_ansible_control_node` so a remote-target test can skip it explicitly instead of failing on an assumption the role was never meant to make in its normal (local) usage.
  5. Installing/running `snapd` itself (squashfs mounts, AppArmor) is generally unreliable inside a lightweight container regardless of distro — confirmed live: even the classic-mount symlink step failed on every distro's test container with `"/snap is not empty, refusing to convert it"` (LXD appears to inject a `/snap` entry into containers on its own). Tagged the whole `snap_packages.yml` (symlink + the snap installs themselves) `requires_virtualization` too.
  6. The `lxd` group (added to in `user_group.yml`) is created by the `lxd` snap package itself, so it doesn't exist once `snap_packages` is skipped — Ubuntu's base images happen to predefine it anyway, but Debian's don't (confirmed live: this task only failed on the Debian test containers, with `"Group lxd does not exist"`). Tagged `requires_virtualization` as well.
  7. Rocky Linux 9 support (added 2026-07-18): a live probe against a throwaway Rocky 9 test container confirmed `ansible.builtin.apt` (used directly for the package-install task) doesn't exist at all on RHEL-family systems — it failed immediately, with Ansible even attempting to auto-install a bogus `python3-apt` dependency first. Fixed by switching to the generic `ansible.builtin.package` module (auto-detects `dnf`) and splitting the package list per OS family (`vars/Debian.yml`, `vars/RedHat.yml` — identical lists except `python3-yaml` is `python3-pyyaml` on RHEL-family), loaded via `include_vars`. Rocky/RHEL/CentOS also need EPEL enabled first (`epel-release`, guarded to exclude Fedora, whose own repos already carry everything needed) for `snapd`/`python3-kubernetes`/`python3-jsonpatch` to resolve. Fedora 43/44 and Rocky 9 declared in `meta/main.yml`; Rocky 10 also declared for forward-compatibility, but has no published LXD image yet (`images:rockylinux` only lists 8 and 9) so it isn't tested live.

  Validated end to end via a dedicated multi-distro smoke test (`ansible/scripts/test_lxd_host_bootstrap_distros.sh` — see [`../scripts/README.md`](../scripts/README.md)): spins up one throwaway LXD container per distro declared in `meta/main.yml` (Ubuntu 24.04/26.04, Debian 12/13, Rocky Linux 9, Fedora 43/44), connects via the `community.general.lxd` connection plugin (`lxc exec`, no SSH — real usage is always `connection: local`, so SSH was never actually part of the role's requirements), and runs the role skipping everything tagged `requires_virtualization`/`requires_ansible_control_node`. **Passes cleanly (`failed=0`) on all seven distros**, covering the genuinely distro-sensitive logic: the SSH keypair task, the package install via `ansible.builtin.package` (confirming `snapd`/`curl`/`python3-kubernetes`/`python3-jsonpatch`/`python3-yaml`(`-pyyaml` on RHEL-family) all resolve on every family, apt or dnf), the EPEL-enablement step (only runs on Rocky, correctly skipped everywhere else including Fedora), and kernel module config persistence. The skipped tasks (snap/LXD install, `lxd_init`, base image import, the `lxd` group, Galaxy collections) remain to be validated for real via `molecule test` with actual `sudo` — see below.
- 🟢 **`k8s_ha_cluster`** (implemented and validated across the full CNI×CSI×topology matrix, 2026-07-18): given a set of already-provisioned/bootstrapped instances, stands up the full HA cluster (kubeadm + kube-vip). Faithfully extracted from `base/02_k8s_base_ha_3_managers_3_workers` (OS prep, containerd, kubeadm tools, kubeadm init/join HA, kube-vip, Headlamp) plus the CSI backends from `base/03`/`base/04` and the Cilium CNI from `base/08` (reused identically by labs 10-14) — all of it already ran for real as part of those labs, so the ported logic is proven. Component selection follows the task-file-indexed-by-name pattern this very document already recommended (`cni: flannel|cilium` → `tasks/cni/<name>.yml`, `csi: none|longhorn|rook_ceph` → `tasks/csi/<name>.yml`), not booleans. Scoped down from the original backlog line by explicit user decision: the Ceph-externo CSI backend is built separately, as `ceph_external_cluster` + `k8s_ceph_external_csi` (see below) rather than a 4th `k8s_ha_cluster_csi` value — it depends on an already-existing external Ceph cluster, the odd one out among the three; node add/remove stays in `k8s_node_scale_cycle` below, not here.
  - Cilium turned out to have real precedent after all (`base/08`'s `kubeProxyReplacement: true` + VIP-pointed `k8sServiceHost`/`k8sServicePort` setup, pinned at `1.19.5`) — an earlier assumption that it would need to be written from scratch was wrong, caught and corrected before implementation.
  - `base/04`'s own `group_vars/all.yml` references `rook_ceph_chart_version`/`ceph_image` in its task file but never defines them — a pre-existing bug in that base lab (left unfixed there, out of scope). This role's own defaults for those two (`v1.20.2`, `quay.io/ceph/ceph:v20.2.2`) were resolved independently via `helm search repo`/`helm show values`.
  - Longhorn's dedicated workload/storage node-pool split (hard-required in `base/03`'s inventory) became optional: both pool groups default to the general workers group unless the caller defines separate ones.
  - Four Molecule scenarios, one per topology (`default`=2m+1w, `ha_3x3`=3m+3w, `single_manager`=1m+0w, `ha_1x2`=1m+2w), each with `k8s_ha_cluster_cni`/`_csi` overridable via `K8S_HA_CLUSTER_TEST_CNI`/`_CSI` env vars — doubling as the topology axis of a full CNI×CSI sweep (`molecule/run_matrix.sh`, 24 combinations, resumable via `k8s_ha_cluster_matrix_results.csv`), on the theory that not every combination is expected to be viable and the sweep exists to find the real limits, not just confirm one happy path.
  - Real bugs found live (via the `default` and `ha_3x3` scenarios so far):
    1. Longhorn's `storage-node=true:NoSchedule` taint was applied unconditionally to the storage pool — with the default (undivided) single-workers-group setup, that tainted the *only* worker, and Longhorn's own `longhorn-ui`/`longhorn-driver-deployer` Deployments don't tolerate that taint (only the `longhorn-manager` DaemonSet does), so they sat `Pending` forever with nowhere left to schedule. Fixed: the taint is now only applied when `k8s_ha_cluster_longhorn_workload_group != k8s_ha_cluster_longhorn_storage_group` (i.e. genuinely dedicated pools).
    2. Idempotence failures on 4 tasks: `kubernetes.core.helm_repository` with `force_update: true` (Longhorn, Rook Ceph ×2, Headlamp) always reports changed, regardless of whether the repo URL actually changed — removed `force_update` from all four (chart versions are pinned explicitly anyway, so nothing depends on force-refreshing the index every run). The join-command/certificate-key "save to local file" tasks always reported changed too, because `kubeadm token create`/`kubeadm init phase upload-certs` genuinely issue fresh, differently-timestamped credentials on every call by design — fixed by only generating/saving them the first time (guarded by whether `join_command.txt` already exists locally), same "already done" pattern used everywhere else in this role.
    3. All 4 scenarios' `verify.yml` used `(nodes_out.stdout | from_json).items` to read the `kubectl get nodes -o json` output's `items` key — but in Jinja2, dot-access on a dict resolves to Python's own `dict.items()` *method* before falling back to a `dict['items']` key lookup, so this failed with `"object of type 'method' has no len()"` the moment `verify` actually ran for real (never caught by `--syntax-check`, which doesn't evaluate Jinja expressions). Fixed by switching to explicit `['items']` bracket access in all 4 files.
  - `ha_3x3` (3 managers + 3 workers) with Cilium + Longhorn + Headlamp passed cleanly end to end (`converge`, `idempotence`, `verify`, `destroy`) after all three fixes above — confirmed live, no leftover VMs afterward.
  - **CNI×CSI matrix sweep: complete, 24/24** (`molecule/run_matrix.sh` — see `k8s_ha_cluster_matrix_results.csv`). **18 pass, 6 `unsupported`-by-design** (not failures — see preflight check below), after 5 real bugs found and fixed live (2026-07-18):
    1. `default`/cilium/longhorn had failed verify: `converge` reported success, but the `longhorn` StorageClass didn't exist yet by the time `verify` checked for it — the role's Longhorn readiness waits only checked `longhorn-manager`/`longhorn-ui` rollout status, not StorageClass creation itself. Fixed: new "Wait for the Longhorn StorageClass to be created" task in `tasks/csi/longhorn.yml` (`until`/`retries` on `kubectl get storageclass longhorn`).
    2. `single_manager`/flannel/none had failed verify: not a role bug, a test-harness timing gap — `verify.yml`'s Ready assertion checked node status once with no `retries`/`wait_for`, and this topology's `converge` finishes right after CNI install (no managers/workers to join afterward) with nothing to naturally give the CNI time to settle, unlike every other topology. Fixed: added `until`/`retries` to the "Get cluster nodes" task in all 4 scenarios' `verify.yml` (same underlying gap, just only manifested here).
    3. `single_manager`/{flannel,cilium}/longhorn had failed at converge: kubeadm taints the control-plane node `NoSchedule` by default, and with no workers at all, that left **no node anywhere** for Longhorn's pods to schedule on (single-node variant of the earlier shared-pool Longhorn taint bug). Fixed: new `tasks/single_node_taint.yml`, wired into `tasks/main.yml` right after CNI install, removes that taint but *only* when both `additional_managers` and `workers` groups are empty — every other topology keeps the taint as kubeadm intends.
    4. **Rook Ceph's CephFS setup was hanging on every topology it was tried on**, waiting forever for a ClientProfile matching `spec.cephFs.subVolumeGroup=="csi"` — root-caused live by holding a hung `ha_3x3`/cilium/rook_ceph cluster up for direct inspection (`kubectl -n rook-ceph get clientprofile -o yaml`) instead of just re-running the sweep: the currently-pinned `ceph-csi-operator`/`rook-ceph` chart versions create exactly **one** `ClientProfile` per cluster (name `rook-ceph`, no per-subvolume-group split at all, and no `subVolumeGroup` field on it whatsoever) — the jsonpath query was filtering on a field that no longer exists, so it could never match. Fixed: `tasks/csi/rook_ceph.yml`'s wait now just grabs `{.items[0].metadata.name}` (whichever ClientProfile exists) instead of filtering by the stale field.
    5. Found immediately after fixing #4 (idempotence failed on the retest): the Rook Ceph cluster values file was written via `copy` then deleted via `file: state: absent` at the end of every run — `copy` is content-idempotent, but a copy+delete cycle always reports changed on both tasks since the file never exists at the start of the next run. Fixed: stopped deleting the file (left in place at `k8s_ha_cluster_rook_values_path`), so the `copy` task now correctly reports unchanged on a second run.
    - `ha_3x3`/cilium/rook_ceph and `ha_3x3`/flannel/rook_ceph — the only two topology/CNI combinations with ≥3 untainted workers — both **pass cleanly end to end** (`converge`/`idempotence`/`verify`/`destroy`) after fixes #4 and #5, confirming both root causes.
    - **Rook Ceph genuinely needs ≥3 worker nodes and nothing less works**, a real (not buggy) limit found while re-testing the remaining combinations after #4/#5: Ceph's `mon.count: 3` (hardcoded in the values) needs 3 nodes without the control-plane taint to reach quorum; mon pods can't schedule on tainted managers, so any topology with fewer than 3 workers never reaches mon quorum, the CephCluster never becomes healthy, and `ceph-csi-operator` never creates a ClientProfile in the first place — this produced log output visually identical to bug #4 before the jsonpath fix, which is why the node-count hypothesis was initially raised, then wrongly dropped, then confirmed correct after all.
    - **Fixed with a fail-fast preflight check** rather than left as a slow, confusing failure: new `tasks/preflight_checks.yml` (wired in as the very first step of `tasks/main.yml`, before `os_prepare`) makes `k8s_ha_cluster_csi=rook_ceph` with fewer than 3 workers an explicitly refused parameterization — `ansible.builtin.fail` with a clear message, in seconds, instead of a 5-13 minute hang ending in a cryptic timeout. `run_matrix.sh` detects this specific failure message and records it as `unsupported` in the CSV, distinct from a genuine `fail_*`/`timeout_*` bug.
    - **Final tally, all 6 topology+CNI combinations with `csi: rook_ceph` and fewer than 3 workers now fail fast as `unsupported`** (~35-60s each, down from 5-13 minutes): `default`/flannel, `default`/cilium, `ha_1x2`/flannel, `ha_1x2`/cilium, `single_manager`/flannel, `single_manager`/cilium.
  - Rocky Linux 9 / Fedora 43/44 support added (2026-07-18), same distro set as `lxd_host_bootstrap`: `containerd.yml`/`k8s_tools.yml` now dispatch to `_debian.yml`/`_redhat.yml` task files by `ansible_facts.os_family` (Docker's separate per-distro repos for Fedora vs Rocky/RHEL/CentOS; `pkgs.k8s.io`'s RPM repo with `disable_excludes: kubernetes`; `dnf versionlock` in place of `dpkg_selections: hold`). `os_prepare.yml` needed no changes (already OS-agnostic). **Design only — not yet live-tested**; `molecule/run_distro_matrix.sh` (new, sweeps all 6 distros against the cheapest `single_manager` scenario, full `molecule test` including `idempotence`) is built and ready to run once the CNI×CSI matrix sweep frees up the host's LXD VM names.
- ✅ **`ceph_external_cluster`** (implemented and validated with real Molecule, 2026-07-18): bootstraps a standalone Ceph cluster via `cephadm` (one monitor + a set of OSD nodes), independent of any Kubernetes cluster. Faithfully extracted from `base/05_k8s_ha_almacenamiento_persistente_externo_ceph`'s `10_desplegar_ceph_externo.yml`. Built as its own role rather than a 4th `k8s_ha_cluster_csi` value — deliberate, per the "odd one out" reasoning already recorded above: this backend needs an entirely separate cluster with its own inventory groups/lifecycle. Pairs with `k8s_ceph_external_csi` below. Generalized the lab's hardcoded "wait for ≥3 OSDs" into "wait for however many OSD hosts the inventory actually declares". Real bugs found live via Molecule's native `idempotence` check:
  1. `ceph orch host add` prints the same "Added host" success message whether the host is genuinely new or already registered — a `changed_when` based on that text was always true. Fixed by checking the current orchestrator host list first (`ceph orch host ls -f json`) and skipping the add entirely for hosts already on it.
  2. Same issue for `ceph orch apply osd --all-available-devices`. First fix attempt used `ceph orch ls --service-type osd -f json` to check for an existing spec, but that output isn't reliably one parseable JSON document (`from_json` failed live with `"Expecting value: line 2 column 1"`, a JSON-lines-shaped quirk, not a JSON array). Fixed by checking the plain-text output of `ceph orch ls --service_name osd.all-available-devices` for Ceph's own `"No services reported"` message instead of parsing JSON.
  - `lxd_machine_provision` extended with an optional per-host `lxd_extra_disk` var (generic, not Ceph-specific) to attach a second raw block device — needed for OSD backing disks, which this role's Molecule scenario uses to provision its 3 OSD VMs.
- ✅ **`k8s_ceph_external_csi`** (implemented and validated with real Molecule, 2026-07-18): wires an already-running Kubernetes cluster to consume an already-bootstrapped `ceph_external_cluster` via `ceph-csi`. Faithfully extracted from `base/05`'s `11_integrar_k8s_ceph_externo.yml`. Verifies with a real PVC write/read through the RBD volume, not just StorageClass existence. Four Molecule scenarios, one per `k8s_ha_cluster` topology (`single_manager`, `default`, `ha_1x2`, `ha_3x3`), keeping the external Ceph cluster side fixed (1 mon + 3 OSD) — `molecule/run_topology_matrix.sh` sweeps all 4 with full `molecule test` each (results in `k8s_ceph_external_csi_topology_matrix_results.csv`). **4/4 pass cleanly** (`converge`/native `idempotence`/`verify`/`destroy`): `single_manager` (570s), `default` (606s), `ha_1x2` (596s), `ha_3x3` (735s, 8 VMs — the most expensive scenario in this repo). No fixes needed for any of the 4 topologies, no leftover VMs or storage volumes after any run — confirms the ceph-csi integration works uniformly regardless of the k8s cluster's size/HA shape.
- ✅ **`db_operator`** (implemented and validated with real Molecule, 2026-07-18): installs a database operator (Percona's PXC/MySQL, PostgreSQL or MongoDB operator, or `mariadb-operator`) into an already-running Kubernetes cluster via Helm — add Helm repo, install chart(s), wait for rollout. Faithfully extracted from the near-identical `12_desplegar_*_operator.yml` step repeated across `base/10_k8s_percona_mysql_pxc`, `base/11_k8s_mariadb_galera`, `base/12_k8s_percona_postgresql` and `base/13_k8s_percona_mongodb`. Originally scoped and named `percona_operator` (3 Percona operators only) per the backlog line below; renamed to `db_operator` and extended with a 4th `mariadb` option mid-implementation, per explicit user request — `mariadb-operator` uses a completely different upstream Helm repo from Percona's, so folding it in under the `percona_operator` name would have been a real mismatch. `pxc`/`postgresql`/`mongodb` share one Helm repo add per task-file (each self-contained); `mariadb` needs its own repo, a separate CRDs chart install, and an extra wait for its validating-webhook Deployment/Service endpoints (its webhook becomes ready a few seconds after the main controller — a `MariaDB` CRD created too early fails with "no route to host"). Version pins taken directly from the base labs' `group_vars/all.yml` (`pxc-operator` 1.20.0, `pg-operator` 3.0.0, `psmdb-operator` 1.22.0, `mariadb-operator` 26.6.0), not guesses. One Molecule scenario installs all four operators in sequence against one cheap k8s node (`csi: none`, no storage needed) — cheaper than one cluster per operator, and each operator's install logic is independent so testing them together doesn't hide cross-operator issues. **Passed cleanly first try** (`converge`/native `idempotence`/`verify`/`destroy`), no bugs found — the pattern was already proven live 4x in the base labs before extraction, leaving little room for surprises. **Per-operator matrix sweep: complete, 4/4 pass** (`molecule/run_matrix.sh` — see `db_operator_matrix_results.csv`): each operator installed *standalone* rather than bundled with the other 3, to confirm none implicitly depends on shared cluster state left behind by another. `pxc` 182s, `postgresql` 181s, `mongodb` 179s, `mariadb` 215s (slowest — the extra CRDs chart + webhook-endpoint wait). No fixes needed for any of the 4, no leftover VMs after any run.
- ⬜ **`generated_secret`**: the repeated "generate a credential if it doesn't already exist, save it to a local file with `0600` permissions, never print it" pattern (PXC, MongoDB, Vault and `vault_admin` passwords...).
- ✅ **`k8s_node_scale_cycle`** (implemented and validated with real Molecule, 2026-07-18): safely adds/removes a node from an already-running `k8s_ha_cluster`-built cluster. Add: provisions the VM via `lxd_machine_provision`, reuses `k8s_ha_cluster`'s own `os_prepare.yml`/`containerd.yml`/`k8s_tools.yml` via relative `import_tasks` (confirmed live to be genuinely self-contained, gated only by `k8s_ha_cluster_nodes_group`), joins with a **freshly minted** `kubeadm token create --print-join-command` rather than reusing `k8s_ha_cluster`'s own `workers_join.yml` (which reads a token cached at initial cluster creation — likely expired by the time a real day-2 scale-up happens), then optionally labels the node into Longhorn. Remove: ports the verified-correct Longhorn `instance-manager` PodDisruptionBudget workaround from `base/10_k8s_percona_mysql_pxc/17_eliminar_nodos.yml` (the only base lab that waits for real replica evacuation instead of a crude `pause: 15s`) — disable Longhorn scheduling on the node, wait for real replicas to evacuate, delete the `instance-manager` pod directly (bypasses its own 0-disruption PDB, safe once no replica remains), then drain/kubeadm-reset/delete/destroy. PXC-specific app-level scale-down (Galera cluster-size reduction, `wsrep_*` health checks) is deliberately out of scope — left to the caller, before invoking this role. One Molecule scenario runs a full add-then-remove cycle against a live 3-node Longhorn-backed cluster (a 4th node added then removed), asserting the original 3 nodes stay `Ready` and untouched throughout. Two real bugs found live, both affecting real callers too, not just the test:
  1. The "add" play's `hosts:` list includes the new node before it exists (this role's own job is to create it), and Ansible's implicit fact gathering tried to connect to it immediately at play start, failing `UNREACHABLE` before the VM-creation task ever ran. Fixed by requiring `gather_facts: false` on that play (documented in the README's example) plus an explicit `ansible.builtin.setup` task scoped to the new node, placed after VM creation and before the OS-dependent task files that need `ansible_facts.os_family`.
  2. `k8s_ha_cluster`'s `k8s_tools_debian.yml`/`k8s_tools_redhat.yml` reference a bare `k8s_major_version` var from that role's own `defaults/main.yml` — invisible here since these task files are imported directly by path rather than via invoking the `k8s_ha_cluster` role itself (whose defaults only merge in when the role is actually invoked). Fixed with this role's own `k8s_node_scale_cycle_k8s_major_version` default (same `"v1.36"` pin, kept in sync manually), passed through explicitly on the `import_tasks` call.
  - Passed cleanly end to end (`converge`/`verify`/`destroy`) after both fixes, no leftover VMs. No native Molecule `idempotence` step for this scenario — `converge` runs a full add-THEN-remove cycle, and both halves perform genuine create/destroy actions by design (provisioning and later destroying the same LXD VM, joining and later deleting the same k8s Node), so every run always reports changed; same one-shot treatment other destructive playbooks in this repo already get.

## Possible improvements (competing roles reviewed)

Similar published roles reviewed: [`juju4/ansible-lxd`](https://galaxy.ansible.com/juju4/lxd),
[`plumelo/ansible-role-lxd`](https://github.com/plumelo/ansible-role-lxd),
[`tideops/ansible-role-kubernetes`](https://github.com/tideops/ansible-role-kubernetes),
[`kubespray`](https://github.com/kubernetes-sigs/kubespray) (feature-by-feature comparison added
2026-07-18, once `k8s_ha_cluster` was actually built),
[`geerlingguy.kubernetes`](https://github.com/geerlingguy/ansible-role-kubernetes).

- **`lxd_host_bootstrap`/`lxd_machine_provision`**: no concrete improvement identified. `juju4/ansible-lxd`
  and `plumelo/ansible-role-lxd` are both more limited than our two roles (they only install LXD and
  networking, with no base-image management, no VM/container selection, no SSH key injection) — nothing
  worth adopting there.
- **`k8s_ha_cluster`**: `tideops/ansible-role-kubernetes` covers a similar scope (kubeadm HA +
  kube-vip + selectable CNI/storage), but uses a simple per-component on/off boolean pattern
  (`install_longhorn: no`, `setup_vip: no`, `install_nginx_ingress: no`), a single un-cataloged CNI
  variable, and only Longhorn as a storage option (no Rook Ceph). Confirmed this remains true after
  building the role for real: our `cni: flannel|cilium` / `csi: none|longhorn|rook_ceph` pattern (each
  loading a distinct `tasks/<axis>/<name>.yml`, not a fixed implementation toggled on/off) stays the
  more flexible design, and Cilium here goes further than tideops documents (real
  `kubeProxyReplacement: true`, not just "configurable CNI").
  - `kubespray` (CNCF project, production multi-cloud scope) is a different category entirely, not a
    real competitor to compare feature-for-feature against a lab-scoped role: 6 CNI options (Calico
    default, Cilium, Flannel, kube-ovn, kube-router, Multus) and cloud-provider CSIs (AWS EBS, Azure,
    Cinder, GCP PD — no Longhorn/Rook Ceph equivalents), dedicated node-scaling playbooks
    (`scale.yml`/`remove-node.yml`, the gap this repo's own planned `k8s_node_scale_cycle` fills for
    labs instead), and by far the broadest distro support of anything reviewed. Confirms the general
    shape of "select CNI/CSI by a single named variable" is the right pattern (matches ours), but at a
    scope this repo isn't trying to reach.
  - `geerlingguy.kubernetes` is deliberately minimal — no HA/kube-vip, no CSI/storage, no ingress, just
    kubeadm + a CNI choice (Flannel/Calico/Weave, explicitly warns against switching after the fact) —
    meant as a building block composed with the author's other roles rather than a full cluster
    installer. Confirms there's no missing baseline feature here either.
  - No concrete design change adopted from this pass — the task-file-indexed-by-name pattern (already
    the only takeaway from the original, pre-implementation review) held up.

## Test plan — Molecule

Using the `default` driver (renamed from `delegated` in Molecule 26.x) against the real host (neither
`lxd_host_bootstrap` nor `lxd_machine_provision` can be tested representatively inside an isolated
Docker/Podman container, since both depend on real virtualization). Each role includes Molecule's
native `idempotence` step in its `test_sequence` — no "always changed" step is accepted as an
exception; for `lxd_machine_provision` this required fixing the SSH key injection to check the existing
content before pushing it, instead of assuming the "always changed" pattern that is accepted elsewhere
in the repo (e.g. Headlamp's `force_update`) — an explicit user decision to require real, verified
idempotency in the new roles. `lxd_machine_provision` has two Molecule scenarios (`default` for VM,
`container` for container — the latter with its own `prepare.yml` to import a container-format image,
since an LXD alias is bound to one format, VM or container, never both).

**Molecule was installed during this session** (`pipx install molecule` + `molecule-plugins[docker]`,
though the driver actually used is `default`/formerly `delegated`, included in core) and `molecule test`
runs for real — not a manual approximation. Adjustments that were needed to make it work:
- `meta/main.yml` needs an explicit `namespace`.
- Each `molecule.yml`'s `provisioner.env.ANSIBLE_ROLES_PATH` must point at
  `${MOLECULE_PROJECT_DIRECTORY}/..` (the parent `ansible/roles/`) so `roles: [...]` resolves without
  installing the role as a collection.

`lxd_host_bootstrap`'s own Molecule scenario is written but still needs a real live `molecule test` run
by the user (it needs `sudo`, unavailable non-interactively in this session) — validated so far via
`--syntax-check`, inspection, and the multi-distro container smoke test described above (which covers
the same OS-package-level logic via a different, `sudo`-free path).
